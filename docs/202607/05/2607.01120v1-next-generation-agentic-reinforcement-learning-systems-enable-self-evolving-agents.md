## 摘要
大语言模型智能体正快速部署到生产环境中，包括编程助手、客户支持聊天机器人和科学研究助手，但在企业部署中它们本质上仍是静态的。大语言模型权重、系统提示、工具库和上下文处理框架在部署时被冻结，任何改进都需要人工策划的数据收集、离线微调、智能体范式修改和重新部署的人工循环。最近关于自进化智能体的研究，例如面向个人用户的OpenClaw，表明智能体能力的下一次飞跃将来自于那些能从自身经验中持续学习的智能体。在本文中，我们认为，阻碍企业级大规模智能体服务实现自进化部署愿景的并非强化学习算法，而是智能体在线强化学习系统。具体而言，当前的智能体强化学习系统及周边可观测性软件栈在三个关键方面存在不足：(i) 缺乏标准化的智能体轨迹数据协议，无法在异构智能体范式中以步骤粒度携带强化学习学习信号；(ii) 缺乏企业级综合数据代理，能将实际工作负载转化为受治理的学习基质；(iii) 缺乏统一的智能体进化控制平面，能基于轨迹统计自动决定何时更新策略权重或进化上下文处理框架。下一代智能体强化学习系统必须围绕这三个支柱进行协同设计，我们勾勒了具体的架构、案例研究和反方论点。通过AReaL2.0，我们实例化了一个分支，将现有的强化学习基础设施重组为面向智能体的在线强化学习循环，用于从已部署的工作负载中更新策略权重。

## Abstract
LLM agents are rapidly being deployed in production, including coding assistants, customer-support chatbots, and scientific research assistants, yet they remain fundamentally static in enterprise deployment. The LLM weights, system prompts, tool repertoires, and in-context harnesses are frozen at deployment time, and any improvement requires a manual loop of human-curated data collection, offline fine-tuning, modification of the agentic paradigm, and re-deployment. Recent work on self-evolving agents, such as OpenClaw for individual users, indicates that the next leap in agent capability will come from agents that continually learn from their own experience. In this paper, we argue that this vision for self-evolving agent deployment is being held back for enterprise-level large-scale agentic service not by reinforcement learning (RL) algorithms but by agentic online RL systems. Specifically, current agentic RL systems and the surrounding observability software stack are inadequate along three essential aspects: (i) there is no standardized agent trajectory data protocol capable of carrying RL learning signals at step granularity across heterogeneous agent paradigms; (ii) there is no enterprise-grade comprehensive data proxy that converts real workloads into governed learning substrates; and (iii) there is no unified agent evolution control plane that automatically decides, based on trajectory statistics, when to update policy weights or evolve the in-context harness. The next generation of agentic RL systems must be co-designed around these three pillars, and we sketch concrete architectures, case studies, and counter-arguments. We instantiate one branch through AReaL2.0, reorganizing existing RL infrastructure into an agent-oriented online RL loop for policy weight updates from deployed workloads.

## 1. 论文的核心问题与整体含义（研究动机和背景）

*   **核心问题**：当前大型语言模型（LLM）智能体在大规模企业级部署中本质上是**静态的**。它们的模型权重、系统提示、工具库等在部署后就被冻结，任何改进都需要依赖昂贵且缓慢的人工循环（手动收集数据、离线微调、重新部署）。这严重阻碍了智能体实现基于自身经验的持续学习和进化，即“自进化”。
*   **研究动机**：为了实现“自进化智能体”这一愿景，论文认为主要瓶颈**并非**在于缺乏更强大的LLM或更有效的强化学习（RL）算法，而在于缺乏一套完整的、面向RL的**系统级基础设施**。当前的可观测性软件栈和智能体RL系统无法将部署后的智能体交互数据高效、安全地转化为可用的学习素材。
*   **整体含义**：论文提出，下一代智能体RL系统必须作为一个系统学科进行设计，而非仅仅作为算法家族。它需要从系统层面解决如何捕获经验、转化为可进行信用分配的数据、选择合适的进化机制并安全部署更新这一系列挑战。

## 2. 论文提出的方法论：核心思想、关键技术细节

论文的核心思想是构建下一代在线智能体RL系统，该系统需要围绕**三个协同设计的支柱**展开：

*   **支柱一：标准化的智能体轨迹数据协议 (ATDP)**
    *   **核心思想**：将智能体经验从普通的、不可学习的日志，转化为类型化的、步骤级别的、可进行RL学习的结构化事件数据。
    *   **关键技术细节**：
        *   **形式化定义**：轨迹 `τ` 由一系列事件 `e_t` 组成，每个事件定义为：`e_t = <o_t, h_t, a_t, y_t, r_t, m_t>`。
            *   `o_t`: 可观测状态（如工具输出、环境状态）。
            *   `h_t`: 隐藏状态（如推理摘要、计划）。
            *   `a_t`: 所选动作（如工具调用、文本生成）。
            *   `y_t`: 动作结果（如工具返回值、用户修正）。
            *   `r_t`: 奖励信号（如二进制结果、标量分数）。
            *   `m_t`: 元数据（如延迟、模型版本、租户）。
        *   **设计原则**：强调**决策相关性**、**跨框架统一**（使学习单元不再是prompt-response对，而是可审计的事件）、**信用可分配性**（能够追溯到具体决策点）、**延迟学习信号支持**（允许后续更新奖励字段）、**版本化可重放性**（记录精确的执行环境以支持重放）和**可治理的观测性**（内置隐私、安全、合规字段）。

*   **支柱二：企业级综合智能体数据代理 (Data Proxy)**
    *   **核心思想**：解决如何在实际的企业异构环境中捕获符合ATDP标准的轨迹数据。它如同一个中间层，负责将生产工作负载转换为受治理的学习素材。
    *   **关键技术细节**：
        *   **框架无关的拦截**：在稳定的执行边界（如LLM API调用、工具调用、记忆读写）进行拦截，不依赖特定智能体框架。
        *   **无损ATDP输出**：将每次拦截的调用无损地转换为ATDP事件流，包括所有学习所需的关键字段。
        *   **可重放能力**：区分确定性重放、近似重放和不可重放事件，使系统能回答“如果使用不同的提示/模型，智能体是否会成功？”。
        *   **跨租户聚合与隔离**：支持基于策略的数据聚合和隔离，满足不同团队的隐私和法律要求。
        *   **奖励信号收集**：从用户交互中自动“收获”各种弱信号和延迟奖励（如用户修正、测试失败、拒绝率等）。

*   **支柱三：统一的智能体进化控制平面 (Evolution Control Plane)**
    *   **核心思想**：解决何时、如何触发进化的问题。将自进化视为一个有治理的决策问题，而非单一的优化器。
    *   **关键技术细节**：
        *   **多表面适应**：智能体包含多个可进化部分（记忆、技能、上下文框架、模型权重、工具模式），控制平面需根据失败类型选择最合适的干预表面。
        *   **自动触发**：基于轨迹统计数据（如评估器得分、用户修正率、工具失败聚类）而非人工审查来自动触发进化。
        *   **算法多元主义**：支持多种RL算法（在线RL、过程奖励学习、蒸馏等），并统一调度。
        *   **安全的分阶段部署**：强制对所有干预进行分阶段评估和部署（如影子评估、回放检查、金丝雀发布）。
        *   **重放优先评估**：在更新应用到用户前，先在回放的历史轨迹上进行测试。
        *   **版本化溯源与回滚**：所有进化的产物（模型、提示、工具模式等）都是版本化的，回滚是进化动作集合中的一等公民。

## 6. 论文的主要结论与发现

*   **主要结论**：企业级自进化智能体的首要瓶颈是缺乏面向在线RL的系统级基础设施，而非算法或模型。下一代智能体RL系统必须围绕**标准化轨迹数据协议**、**企业级数据代理**和**统一进化控制平面**这三个支柱进行协同设计。
*   **核心发现**：
    1.  只有将智能体经验转化为可信用分配、可重放的结构化数据，才能进行有效的在线学习。
    2.  生产环境中的异构智能体框架需要一个框架无关的代理层来捕获和治理数据。
    3.  自进化不等于盲目更新模型权重，应是一个包含多种干预手段并受治理的决策过程。
    4.  AReaL 2.0的实例表明，将现有的RL基础设施重新组织为服务组件，以支撑在线策略更新是技术上可行的。
